¿Qué es un control de versiones, y por qué debería importarte?

Un control de versiones es un sistema que registra los cambios realizados en un archivo o conjunto de archivos a lo largo del tiempo, de modo que puedas recuperar versiones específicas más adelante. Aunque en los ejemplos de este curso usarás archivos de texto, en realidad puedes hacer lo mismo con casi cualquier tipo de archivo que encuentres en una computadora.

Un sistema de control de versiones te permite regresar a versiones anteriores de tus archivos, regresar a una versión anterior del proyecto completo, comparar cambios a lo largo del tiempo, ver quién modificó por última vez algo que pueda estar causando problemas, ver quién introdujo un problema y cuándo, y mucho más. Usar un sistema de control de versiones también significa generalmente que si arruinas o pierdes archivos, será posible recuperarlos fácilmente.

Métodos de control de versiones

Sistemas de Control de Versiones Locales

Un método de control de versiones común es ir creando versiones de un archivo manualmente, por ejemplo guardar una copia de la versión 1 y luego seguir trabajando con la copia de la versión 2 y así.

Este método es común porque es muy sencillo, pero también es propenso a errores. Puedes llegar a borrar o sobrescribir archivos que no querías. Otra forma de guardar cambios bastante fea podría ser comentar bloques de código para deshabilitar ciertas funciones sin eliminarlo por si llegas a necesitarlo después.

Para afrontar estos problema los programadores desarrollaron hace tiempo sistemas locales que contenían una simple base de datos, en la que se llevaba el registro de todos los cambios realizados a los archivos. En este punto el control de versiones se llevaba a cabo en el computador de cada uno de los desarrolladores y no existía una manera eficiente de compartir el código entre ellos.

Sistemas de Control de Versiones Centralizados

Para facilitar la colaboración de múltiples desarrolladores en un solo proyecto los sistemas de control de versiones evolucionaron: en vez de almacenar los cambios y versiones en el disco duro de los desarrolladores, estos se almacenaban en un servidor centralizado.

Esta configuración también tiene serias desventajas. La más obvia es el punto único de fallo que representa el servidor centralizado. Si ese servidor se cae durante una hora, entonces durante esa hora nadie podrá colaborar o guardar cambios. Si el disco duro en el que se encuentra la base de datos central se corrompe, y no se han realizado copias de seguridad adecuadamente, se perderá toda la historia del proyecto, con excepción de las copias instantáneas que las personas tengan en sus máquinas locales. Los Sistemas de Control de Versiones Locales también sufren de este mismo problema: cuando tienes toda la historia del proyecto en un mismo lugar, te arriesgas a perderlo todo.

Sistemas de Control de Versiones Distribuidos

En un Sistema de Control de Versiones Distribuido, como por ejemplo Git, los clientes tienen una copia completa del proyecto con todas sus versiones incluidas. De esta manera, si un servidor deja de funcionar cualquiera de los repositorios disponibles en los clientes puede ser copiado al servidor con el fin de restaurarlo.

Las Tres Secciones Principales

Las tres secciones principales de un proyecto de Git son el Directorio de Trabajo (Working Directory), el Área de Preparación (Staging Area) y Directorio de Git o Repositorio (Git Directory or Repository).

  • Working Directory o directorio de trabajo

    Es el directorio o carpeta donde se encuentra la copia de una versión del proyecto sacada de la base de datos del Git Directory. Es como tu área de trabajo donde puedes agregar, editar y eliminar los archivos.

  • Git Directory/Repository o directorio Git/repositorio

    Es la carpeta oculta (.git) que se crea con el comando git init. En el mismo generalmente se encuentran dos lugares importantes que son el Staging Area y el historial de commits, o sea, la "base de datos" de tu proyecto donde se irán guardando las versiones del mismo. El nombre del directorio comienza con "." para indicar que es un archivo oculto. El sistema operativo por defecto no muestra los archivos ocultos al usuario.

  • Staging Area o área de preparación (a veces referido como "index")

    Es un archivo que almacena información acerca de lo que va a ir en tu próxima confirmación (commit). Es una especie de borrador y cada vez que termines de trabajar en un archivo del Working Directory, querrás COPIARLO en el Staging Area (usando el comando git add). Parece innecesario pero te puede llegar a dar cierta flexibilidad y orden en los cambios porque nos permite elegir que y cuando "commitear". Por ejemplo en vez de hacer:

    git commit -m "Add HTML and CSS"

    Podemos hacer:

    git commit -m "Add HTML"

    git add style.css

    git commit -m "Add CSS"

    Luego en el historial aparecería por separado lo que puede dar más orden. No solo podemos agregar archivos al Staging Area, también los podemos quitar.

El flujo de Trabajo Básico:

  1. Agregas, modificas, eliminas archivos en tu Working Directory
  2. Mandas una copia de los mismos al Staging Area
  3. Haces un commit para que queden guardados como una versión de tu proyecto en el Git Directory o Repositorio

Los Estados de Git

  1. Untracked (sin seguimiento): los archivos sin seguimiento se han creado dentro del Working Directory pero aún no se han agregado al Staging Area mediante el comando git add. Son cualquier archivo en tu Working Directory que no estuvo en tu último commit ni está en el Staging Area.
  2. Unmodified (sin modificar): se hizo un commit al archivo y no se volvió a modificar.
  3. Modified (modificado): el archivo tuvo cambios pero NO fue COPIADO al Staging Area.
  4. Staged (preparado): el archivo fue COPIADO al Staging Area.
  5. Committed (confirmado): una versión concreta de un archivo está guardada en el Git Directory.

¿Cómo Git guarda los datos?

Git toma "instatáneas", "snapshots" o "fotos" de tu proyecto al momento de mandarlo al Git Directory/Repository con el comando git commit. No solo considera el archivo al que le hiciste el cambio sino que considera a todos ellos en ese momento. Si una versión de tu proyecto se conforma de muchos archivos es más sencillo usar Git para guardar esa versión, tomándole una "foto" en ese momento, que si lo estuvieramos haciendo manualmente. Esta manera de trabajar en mucho más fácil, segura y eficiente. Tomaría mucho trabajo estar copiando y pegando cientos de archivos manualmente en vez de solo usar Git para sacarle una "foto" de como se encuentra en ese momento y listo. Luego más adelante podríamos volver a ese punto exacto si lo necesitáramos. Para ser eficiente, si los archivos no se han modificado Git no almacena el archivo de nuevo, sino que lo enlaza con la última copia que ya tiene almacenada.
