Comandos Básicos (entorno local)

  • git status: muestra el estado del Working Directory y el Staging Area. Permite ver qué cambios se han realizado, cuáles no y qué archivos no están siendo rastreados por Git.
  • git add : agrega un cambio del Working Directory al Staging Area. Le dice a Git que desea incluir algún archivo en el próximo commit. Los cambios no se registran hasta que ejecutas git commit.
  • git restore: eliminar los cambios que hayamos hecho en el Working Directory y volver al estado en el que estaba en la última confirmación (o cuando estaba recién clonado, o como sea que haya llegado a tu directorio de trabajo).
  • git commit -m "mensaje descriptivo": captura una "instantánea" de los cambios realizados actualmente en el proyecto. Las instantáneas son como versiones "seguras" de un proyecto: Git nunca las cambiará a menos que se lo solicite explícitamente.
  • git log: ver el histórico de commits.
  • git rm --cached: eliminar archivos del Staging Area.
  • git diff: ver la diferencia entre lo que hay en el Working Directory y el Staging Area o en el Git Directory en caso que no haya nada en el Staging Area.
  • git diff --staged: ver diferencia entre Staging Area y Git Directory.

Comandos Básicos de la Consola (recomendado aprender):

  • pwd (print working directory): imprime la ruta del directorio de trabajo actual, o sea te dice donde estás parado.
  • ls -a (list all): muestra todos los archivos y directorios incluyendo a los que están ocultos.
  • cd [nombre-subdirectorio] (change directory): te mueve al subdirectorio especificado. Básicamente lo usas para moverte entre carpetas.
  • cd ..: moverte hacia atrás.
  • mkdir [nombre-directorio] (make directory): crear un directorio (carpeta) nuevo.
  • touch [nombre-archivo]: crear un archivo nuevo. Ejemplo: touch notas.txt para crear un archivo de texto llamado "notas".
  • cat [nombre-archivo]: leer el contenido de un archivo. Ej: cat notas.txt (te mostrará su contenido suponiendo que no esté vacío).
  • cat > [nombre-archivo]: escribir dentro de un archivo desde la misma consola. Para guardar y salir del modo escritura presiona Ctrl + D.

Podemos empezar a usar Git de dos maneras. La primera es inicializando un Git Directory/Repository en un directorio o carpeta existente. La segunda es clonar un repositorio existente que use Git desde otro servidor (se verá más adelante).

En esta sección usaremos la Inicialización. Vamos a crear un directorio llamado "Proyecto" en el escritorio y vamos a navegar desde nuestra consola Git Bash hasta dentro de esa carpeta. Podemos hacerlo fácilmente con el comando cd y arrastrando la carpeta a la consola o bien hacemos click derecho sobre el directorio y luego en "abrir con Git Bash".

cd (arrastramos la carpeta a la consola)

Una vez dentro del directorio "Proyecto" desde nuestra consola vamos a inicializar un Git Directory/Repository con el comando:

git init

Luego creamos un archivo de texto llamado "notas.txt". El repositorio de Git no sabe que existe el archivo "notas.txt" porque está en estado Untracked. Podemos comprobarlo con el comando:

git status

Este comando te brinda la siguiente información: la rama en donde te encuentras (por defecto es la main o master), que todavía no se han hecho commits y que tenemos un archivo Untracked. También nos menciona que para empezar a hacerle seguimiento debemos escribir un comando, hagámoslo:

git add notas.txt

Ahora Git COPIÓ nuestro archivo al Staging Area, se encuentra en un estado de Staged y está esperando para ser trasladado al Git Directory.

Si ponemos de nuevo el comando git status ahora nos dirá que tenemos el archivo esperando a ser Committed y que podemos usar el siguiente comando para que vuelva a estar Untracked:

git rm --cached [file]

Este comando puede ser muy útil por ejemplo si olvidaste añadir algo en tu archivo .gitignore y lo mandaste al Staging Area accidentalmente.

Ahora para mandar el archivo al Git Directory (sacarle una "foto" en su estado actual) y que quede guardado en la "base de datos" vamos a usar el comando:

git commit -m "mensaje descriptivo"

Entre las comillas debemos comentar que cambios realizamos. Más adelante explicaré como debería ser redactado. Ahora nuestro archivo se encuentra en el Git Directory guardado en un estado de Committed y Unmodified. Si escribimos de nuevo git status nos dirá que nuestro arbol de trabajo está limpio porque no tenemos nada rastraeado y modificado en el Working Directory ni en Staging Area.

Después de haber hecho alguna confirmación, o si has clonado un repositorio que ya tenía un histórico de confirmaciones, quizás quieras mirar el historial de confirmaciones con:

git log notas.txt

Ahí nos muestra datos como el hash (números y letras que identifican a un commit en específico), la rama (con un HEAD que es como un puntero que te indica que te encuentras en la rama main en este caso), el autor, la fecha y el mensaje del commit. Si introduces el comando git log sin más te muestra el historial de "fotos" del Repository entero. Puedes por ejemplo visualizar los "parches" o cambios de las dos últimas "fotos" con git log -p -2 o ver estadísticas con git log --stat, entre otras opciones y filtros muy útiles.

Si ahora modificamos el archivo de texto agregándole algún contenido y guardamos, el estado del archivo cambiaría a Modified. Podemos comprobarlo nuevamente con el comando:

git status

Nos menciona que usemos git add para copiar el archivo al Staging Area y que también podemos usar git restore.

Añadamos ahora esta modificación al Staging Area con git add, que por cierto no mueve el archivo de un lado a otro, sino que nos crea una copia del mismo dentro del Staging Area mientras que la otra copia se mantendrá en el Working Directory. A continuación te mostraré porqué es importante esta distinción.

git add notas.txt

Ahora si modificamos de nuevo el archivo de texto y escribimos git status nos dirá que nuestro archivo está Staged y la vez nos dice que está MODIFIED y NO STAGED. Por eso es importante la distinción. En el Staging Area se encuentra la copia del archivo con la primera modificación y ahora que modificamos el archivo de nuevo, este difiere de lo que hay en el Working Directory con lo que hay en el Staging Area. Entonces si modificas un archivo luego de ejecutar git add, deberás ejecutar git add de nuevo para preparar la última versión del archivo.

También nos mencionan que podemos eliminar el archivo del Staging Area usando el comando:

git rm --cached notas.txt

Para ver la diferencia entre lo que está en el Staging Area y en el Working Directory usamos el comando:

git diff

Diferencia la versión vieja con la versión nueva, nos muestra los índices donde están guardados, cuantas líneas se quitaron y agregaron (@@ -1 +1,2 @@ significa que se quitó 1 línea a partir de la línea 1 y se agregaron 2 líneas a partir de la línea 1, en una diferencia más realista verías un encabezado como: @@ -34,6 +34,8 @@ en este ejemplo se extrajeron 6 líneas a partir de la línea número 34 y se agregaron 8 líneas a partir de la línea número 34) y finalmente cuales fueron las palabras que eliminamos y agregamos.

Haré un git add y git commit a la última versión del archivo. Ahora cambiaré completamente el contenido del archivo de texto con un "Hola mundo" y lo mandaré al Staging Area de nuevo. Para ver la diferencia entre la última foto (commit) y lo que tenemos ahora mismo en el Staging Area podemos usar el comando:

git diff --staged

Como verás git diff nos ayuda a responder dos preguntas: ¿Qué cambié pero no preparé? y ¿Qué preparé y está listo para confirmar?
